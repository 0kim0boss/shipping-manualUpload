# 배송사별 송장 업로드 양식 변환 페이지 구현계획서

작성일: 2026-06-24

## 1. 목표

배송 raw data 단일 시트를 업로드하면 주문번호 기준으로 주문을 리스트업하고, 각 주문별 운송사 선택값에 따라 FEDEX, DHL, 우체국, ACI 업로드용 파일로 변환하여 다운로드할 수 있는 GitHub Pages 배포용 웹 페이지를 구현한다.

## 2. 전제 및 현재 확인 상태

| 항목 | 가능 여부 | 코멘트 |
| --- | --- | --- |
| GitHub Pages 배포 | 가능 | 정적 웹앱으로 구현 가능하다. React/Vite 또는 단일 HTML + JS 방식 모두 가능하다. |
| 로컬 샘플 및 기존 코드 확인 | 가능 | `sample` 폴더에서 FEDEX `.xlsm`/`.in`, DHL `.csv`, ACI US `.xlsx`, raw data `.xlsx`, 우체국 K-packet 생성 코드(`index.html`)를 확인했다. |
| 브라우저에서 raw data 업로드 및 파싱 | 가능 | `.xlsx`, `.xls`, `.csv` 파일 업로드 후 브라우저 메모리에서 파싱 가능하다. 서버 업로드 없이 처리할 수 있다. |
| 주문번호 기준 리스트업 | 가능 | raw data의 주문번호 컬럼을 지정하거나 자동 매핑 규칙을 두면 가능하다. 중복 주문번호가 있을 경우 묶음 처리 정책이 필요하다. |
| 주문별 운송사 드롭다운 선택 | 가능 | 화면에서 각 주문 행마다 운송사를 선택하게 구현 가능하다. 일괄 선택 기능도 추가 가능하다. |
| Generate 후 배송사별 파일 생성 | 가능 | DHL CSV, 우체국 K-packet Excel, ACI Excel, FEDEX `.in` 모두 브라우저에서 1차 생성 가능하다. |
| 수출자 정보 고정값 + 커스텀 메뉴 | 가능 | 기본값을 앱 설정에 내장하고, 사용자가 화면에서 수정 가능하게 만들 수 있다. 저장은 브라우저 `localStorage`에 가능하다. |
| Dark mode + blue/purple button style | 가능 | GitHub Pages 정적 UI에서 문제 없이 구현 가능하다. |

## 2.1 샘플 파일 분석 결과

| 파일 | 확인 결과 | 가능 여부 | 코멘트 |
| --- | --- | --- | --- |
| `sample/raw data sample.xlsx` | `Products` 단일 시트, 범위 `A1:V15` | 가능 | 주문 기준 컬럼은 `Order id`, 주문 표시용 번호는 `Order no`로 보인다. 동일 `Order id`가 여러 행이면 다품목 주문으로 묶어야 한다. |
| `sample/dhl sample.csv` | CSV 단일 파일 | 가능 | 헤더가 존재하며 raw data와 유사한 필드 구조다. DHL 출력은 헤더 순서 유지 후 CSV 생성 방식으로 구현 가능하다. |
| `sample/aci sample_US.xlsx` | `Form`, `국가명_영문명`, `참고사항` 3개 시트 | 가능 | 설명상 단일 시트라고 했지만 실제 샘플은 참조 시트를 포함한다. 출력 시 `Form`에 값을 쓰고 나머지 참조 시트는 보존하는 방식이 적합하다. |
| `sample/fedex sample.xlsm` | 13개 시트와 `xl/vbaProject.bin` 포함 | 가능/주의 | 매크로 포함 파일이다. 실제 입력 핵심은 `020 Transaction`, 기본값은 `기본값(필수 입력)`, 필드 정의는 `Field#`로 보인다. |
| `sample/fedex sample.in` | CSV text, `fieldNumber,"value"` 반복 구조 | 가능성 높음 | `.in`은 바이너리가 아니라 텍스트 포맷이다. 샘플은 4개 주문 레코드로 보이며, FEDEX `.in` 직접 생성 재구현 가능성이 높다. |
| `sample/code.rtf`, `sample/index.html` | 기존 Apps Script/프론트엔드 코드 | 가능 | 우체국 K-packet 엑셀 생성 로직이 `index.html`의 `buildKoreaPostRow`, `downloadKoreaPostExcel`에 구현되어 있다. |

## 3. 권장 기술 구성

| 항목 | 제안 | 가능 여부 | 코멘트 |
| --- | --- | --- | --- |
| 배포 | GitHub Pages | 가능 | 정적 파일만 배포하므로 운영 비용이 없다. |
| 프론트엔드 | Vite + React + TypeScript | 가능 | 주문 리스트, 드롭다운, 설정 메뉴 등 상태가 있는 UI에 적합하다. |
| 엑셀/CSV 파싱 | SheetJS 또는 ExcelJS + PapaParse | 가능 | `.xlsx`/`.csv` 처리 가능. `.xlsm` 매크로 보존/실행은 별도 검토가 필요하다. |
| 파일 다운로드 | JSZip + FileSaver 또는 브라우저 Blob API | 가능 | 여러 배송사 출력물을 zip으로 묶어 다운로드 가능하다. |
| 설정 저장 | localStorage | 가능 | 수출자 정보, 최근 컬럼 매핑, 기본 운송사 등을 브라우저에 저장 가능하다. |

## 5. 배송사별 변환 계획

### 5.1 DHL

| 항목 | 가능 여부 | 코멘트 |
| --- | --- | --- |
| CSV 단일 시트 생성 | 가능 | 브라우저에서 CSV 생성 가능하다. |
| 템플릿 컬럼 순서 반영 | 가능 | DHL 샘플 CSV를 프로젝트에 넣으면 컬럼 순서와 헤더명을 그대로 매핑한다. |
| 인코딩 처리 | 가능 | DHL 홈페이지 요구사항에 따라 UTF-8 BOM 또는 EUC-KR 여부를 확인해야 한다. |

권장 방식: DHL 템플릿 CSV 헤더를 기준으로 raw data와 수출자 정보를 매핑하여 `.csv` 파일을 생성한다.

### 5.2 우체국

| 항목 | 가능 여부 | 코멘트 |
| --- | --- | --- |
| Excel 단일 시트 생성 | 가능 | 기존 `index.html` 코드가 SheetJS로 `우체국등록` 시트를 생성한다. |
| 기존 양식 스타일 유지 | 제한적 | 기존 코드는 템플릿 스타일 보존이 아니라 AOA 배열로 새 workbook을 만드는 방식이다. 업로드용 데이터 파일 목적이면 충분하지만, 공식 양식의 시각 스타일 보존은 별도 템플릿이 필요하다. |
| 업로드용 파일 다운로드 | 가능 | `.xlsx` 파일로 다운로드 가능하다. |
| K-packet 자동 분류 | 가능/주의 | 파일 업로드 경로는 `K-PACKET`, `우체국`, `KOREA POST`를 우체국으로 인식한다. 기존 Google Sheet 서버 로딩 경로는 `우체국`만 인식하므로 `K-packet`도 인식하도록 보강해야 한다. |

권장 방식: 기존 `buildKoreaPostRow(order, productType)`의 65컬럼 매핑을 TypeScript로 이식하고, SheetJS 또는 ExcelJS로 `우체국등록` 단일 시트 `.xlsx`를 생성한다.

샘플 코드 기준 우체국 생성 규칙:

| 항목 | 구현 내용 |
| --- | --- |
| 상품구분 | 다운로드 전 모달에서 `Gift` 또는 `Merchandise` 선택 |
| 출력 시트명 | `우체국등록` |
| 파일명 | `KoreaPost_Bulk_YYYY-MM-DD.xlsx` |
| 헤더 | 1행 `우체국 대량등록 양식`, 2행 컬럼명, 3~5행 빈 행, 6행부터 주문 데이터 |
| 품목 다중값 | 상품명, 수량, 가격, 순중량, HS CODE, 생산지를 세미콜론(`;`)으로 연결 |
| 기본 HS CODE | `3304991000` |
| 기본 생산지 | `KR` |
| 기본 순중량 | 품목별 `30`g |
| 박스 실중량 | `package.weight` kg 값을 g으로 변환 |
| EMS구분/물품종류 | CL/MX 국가는 `R`/`re`, 그 외 국가는 `K`/`rl` |
| 보험가입여부 | `N` |
| 고정 사업자번호 | `1278701611` |
| 고정 발송자 주소 | `611-612, 26, Beotkkot-ro 24-gil, Geumcheon-gu, Seoul, Republic of Korea, 08517` |
| 박스 치수 | 63~65번째 컬럼에 length/width/height 반올림 값 입력 |

### 5.4 FEDEX

| 항목 | 가능 여부 | 코멘트 |
| --- | --- | --- |
| 입력 시트 값 생성 | 가능 | 샘플 기준 핵심 입력 시트는 `020 Transaction`이다. 1행은 필드명, 3행은 FEDEX field number, 4행부터 주문 데이터가 들어간다. |
| 브라우저에서 Excel 매크로 실행 | 불가 | GitHub Pages/브라우저는 VBA 매크로를 실행할 수 없다. 보안상 Excel 매크로 실행 환경이 없다. |
| FEDEX 템플릿의 매크로를 코드에 그대로 삽입 | 제한적/대체 필요 | VBA를 JS에 그대로 삽입해 실행하는 것은 불가능하다. 매크로가 생성하는 `.in` 파일 포맷을 분석해 JS로 동일 로직을 재구현해야 한다. |
| `.in` 파일 직접 생성 | 가능성 높음 | 샘플 `.in`이 CSV text이며 `fieldNumber,"value"` 반복 구조다. `020 Transaction` 3행의 field number와 기본값 시트의 field number를 조합해 직접 생성 가능해 보인다. |
| `.xlsm` 매크로 보존 후 다운로드 | 조건부 가능 | 일부 라이브러리/상용 라이브러리는 매크로 보존이 가능할 수 있으나, 실행은 사용자의 로컬 Excel에서 해야 한다. 일반적인 무료 브라우저 라이브러리는 매크로 보존이 제한적이다. |

FEDEX 가능한 방식:

1. 최선안: 샘플 `.xlsm`의 `기본값(필수 입력)` 시트와 `020 Transaction` 시트의 field number를 읽어 `.in` 파일을 TypeScript로 직접 생성한다.
2. 대안 A: 브라우저에서 FEDEX `.xlsm` 입력 시트까지만 채운 파일을 다운로드하고, 사용자가 로컬 Excel에서 매크로 버튼을 눌러 `.in` 파일을 생성한다.
3. 대안 B: GitHub Pages가 아닌 백엔드 서버를 추가하고, 서버에서 변환을 수행한다. 단, Linux 서버에서 VBA 매크로 실행은 일반적으로 어렵기 때문에 `.in` 포맷 재구현이 여전히 가장 안정적이다.
4. 대안 C: Windows PC에 Excel 자동화 전용 로컬 변환 도구를 별도로 두고 웹앱은 입력 파일 생성까지만 담당한다. 운영 편의성은 떨어진다.

샘플 기준 FEDEX `.in` 생성 구조:

| 원천 | 역할 |
| --- | --- |
| `기본값(필수 입력)` 시트 | 발송자 회사명, 주소, 연락처, 통화, 무게단위 등 모든 주문에 반복되는 field number/value 제공 |
| `020 Transaction` 시트 1행 | 사용자 화면과 매핑표에서 보여줄 사람이 읽는 필드명 |
| `020 Transaction` 시트 3행 | `.in`에 기록할 FEDEX field number 제공 |
| `020 Transaction` 시트 4행 이후 | 주문별 수취인/품목/무게/박스/서비스 정보 |
| `Grouping Fields` 시트 | 같은 주문번호의 복수 품목 처리 시 반복 필드 후보로 사용 |

```text
  aci/US.xlsx
  aci/JP.xlsx
  fedex.xlsm
```

## 8. GitHub Pages 기준 제약

| 제약 | 영향 | 가능한 방식 |
| --- | --- | --- |
| 서버 없음 | 파일은 사용자 브라우저 안에서만 처리된다. | 개인정보 보호 측면에는 유리하다. 모든 변환 로직을 프론트엔드 JS로 구현한다. |
| VBA 매크로 실행 불가 | FEDEX `.in` 자동 생성이 막힐 수 있다. | `.in` 포맷을 분석해 JS로 재구현하거나, 사용자가 로컬 Excel에서 매크로를 실행하게 한다. |
| 비밀키/민감정보 저장 불가 | API 연동이나 인증형 기능에 부적합하다. | 이번 범위에서는 외부 API 없이 구현한다. |
| 큰 파일 처리 성능 제한 | 대량 주문 파일에서 브라우저가 느려질 수 있다. | Web Worker 적용, 미리보기 행 제한, Generate 중 진행률 표시로 완화한다. |

## 10. FEDEX 매크로 확인에 필요한 파일

FEDEX 자동 `.in` 생성 가능 여부를 판단하려면 아래 파일이 필요하다.

| 필요 파일 | 이유 |
| --- | --- |
| FEDEX 원본 `.xlsm` 템플릿 | 확인 완료. 시트 구조와 매크로 존재 확인 |
| 매크로 실행 전 샘플 입력값 | raw data 샘플 및 `.xlsm` 입력 행으로 일부 확인 |
| 매크로 실행 후 생성된 `.in` 샘플 | 확인 완료. CSV text 구조 |
| 추가 `.in` 샘플 | 다품목 주문, US 외 국가, 전화번호/우편번호/주 코드 예외 검증용 |
| 가능하면 VBA 코드 접근 권한 | 필수는 아니지만, 매크로의 예외 처리 규칙 확인에 도움 |

판단 기준:

- 현재 샘플 기준: FEDEX `.in` 직접 생성 가능성이 높다.
- 남은 리스크: VBA 내부의 숨은 예외 처리, 국가별 필수값, 다품목 그룹핑 방식, `#ERROR!` 값 처리 규칙.
- VBA 코드 접근 불가 시에도 `Field#`, `Grouping Fields`, `.in` 샘플을 기준으로 1차 구현은 가능하다.

## 11. 최종 권장 범위

1차 배포 범위:

1. raw data 업로드
2. 주문번호 기준 리스트업
3. 주문별 운송사 선택
4. 수출자 정보 설정
5. DHL CSV 생성
6. ACI US XLSX 생성
7. FEDEX `.in` 직접 생성 1차 구현
8. 우체국 K-packet XLSX 생성
9. zip 다운로드
10. GitHub Pages 배포

FEDEX는 여전히 이 프로젝트의 핵심 리스크지만, 샘플 `.in`이 텍스트 구조로 확인되었기 때문에 1차 구현 범위에 포함할 수 있다. GitHub Pages 자체가 매크로를 실행할 수 없으므로, “매크로를 코드에 삽입”하는 방식이 아니라 “`.xlsm`의 field number와 샘플 `.in` 구조를 기준으로 TypeScript에서 직접 생성”하는 방식으로 진행한다.
